# Zoho Books API Payload Audit & Schema Comparison

This document provides a detailed comparison and gap analysis between the write payloads generated by the Flutter application (defined in [zoho_write_payloads.json](file:///E:/work/nellon/zoho_write_payloads.json) and compiled from models inside `lib/data/models/`) and the official Zoho Books OpenAPI specification definitions (defined inside the [zohodocs](file:///E:/work/nellon/zohodocs) directory).

---

## 1. Executive Summary

The Flutter application acts as an offline-first client syncing transactions with Zoho Books. Currently, the application operates in transaction mock mode (`_mockTransactions = true`), routing database writes to simulated sandboxes. 

While the generated payloads contain the appropriate business parameters (contacts, invoices, payments, etc.), a strict comparison against the official API schemas reveals that:
1. **Unrecognized Meta-Fields**: The app sends local operational attributes (like internal database IDs, sync status flags, and route indices) that Zoho's API doesn't specify in its OpenAPI docs. Although standard REST endpoints generally ignore unrecognized fields, these are technically non-standard.
2. **Nested Model Serialization**: The app serializes nested domain data (like complete item objects under invoice/order line items) instead of just the primitive parameters Zoho expects.
3. **Critical Expense Sync Failure**: The Expense synchronization structure contains a major mismatch. The app sends a nested lines array, whereas Zoho Books expects a single flat entry at the root. More importantly, the required field `paid_through_account_id` is missing, which will cause API failures if mock mode is turned off.

#### Gap Analysis & Impact
The standard Zoho Books `/expenses` endpoint does not support a nested items list (`lines`). It is designed for posting single-item expenses. 
1. **Missing Required Fields at Root**: The Zoho API requires `account_id` and `paid_through_account_id` directly in the root JSON block. The app's payload lacks `paid_through_account_id` completely, and `account_id` is only defined inside the nested `lines` items.
2. **Structural Mismatch**: The app groups expense amounts under a `lines` array. The standard Zoho Books API does not expect a `lines` array for basic expenses.
* **Severity**: **High (Critical)**. This payload will fail against the live Zoho Books API. A conversion helper is required to flatten this structure before sending it to Zoho.

## 3. Actionable Recommendations

To prepare the application for production (exiting sandbox mock mode), implement the following adjustments:

### 1. Flatten the Expense Payload
Ensure the transaction payload is flattened to match the single-expense API structure before sending. 

Modify the client or worker to flatten the model array:
```dart
// Example correction for Expenses in ZohoApiClient:
final flattenedExpense = {
  'date': expenseJson['date'],
  'amount': expenseJson['amount'], // Summed total
  'location_id': expenseJson['location_id'],
  'account_id': expenseJson['lines'].isNotEmpty ? expenseJson['lines'].first['account_id'] : 'general_expense_ac_id',
  'paid_through_account_id': 'acc_cash', // Map to local cash/bank account ID
  'description': expenseJson['lines'].map((l) => l['description']).join(', '),
};
```

### 2. Clean Up Payloads (Optional)
Remove local tracking fields (e.g., `isPendingSync`, `customer_name`, `invoice_id`, nested `item` maps) before sending the request to the server to reduce payload size.
